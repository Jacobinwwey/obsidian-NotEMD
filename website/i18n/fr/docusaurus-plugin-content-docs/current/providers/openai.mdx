# OpenAI Fournisseur

<TLDR>
**OpenAI utilise le transport partagé compatible avec OpenAI.** La configuration par défaut actuelle prend en compte `gpt-4o`, `https://api.openai.com/v1`, `temperature: 0.5` ainsi que le test de connexion `models-then-chat`. Utilisez cette page lorsque vous avez besoin d’une sémantique précise des champs ; consultez l’aperçu des [LLM Fournisseurs](/docs/providers/overview) pour effectuer une sélection au niveau des catégories.

Ceci fait partie du [Obsidian Guide de gestion des connaissances IA](/docs/pillar-ai-knowledge).
</TLDR>

## Installation

Créez une clé OpenAI API, ajoutez un profil de fournisseur `OpenAI` dans les paramètres Notemd, et conservez la valeur par défaut de Base URL à moins que vous ne visiez intentionnellement un gateway compatible avec OpenAI. Le champ modèle peut rester sur `gpt-4o` pour des utilisations générales ou être remplacé par un autre ID de modèle OpenAI que votre compte peut utiliser.

Utilisez des modèles spécifiques à tâche pour le contrôle des coûts : conservez des modèles plus puissants pour la recherche, l’extraction de concepts et la restructuration de longs textes ; utilisez des modèles moins chers ou plus rapides pour la traduction, les suggestions de liens et les résumés courts.

## Point de terminaison et authentification

| Champ | Préférence actuelle |
|-------|----------------|
| Transport | `openai-compatible` |
| Clé API | Obligatoire |
| Base URL | `https://api.openai.com/v1` |
| Modèle par défaut | `gpt-4o` |
| Température | `0.5` |
| Test de connexion | `/models`, puis `/chat/completions` |

Notemd normalise les bases URL compatibles avec OpenAI avant d’ajouter `chat/completions` ou `models`. Ne pas inclure de `/chat/completions` à la fin de la base URL ; cela crée un chemin dupliqué.

## Découverte de modèles

OpenAI utilise le chemin de découverte de modèles générique compatible avec OpenAI. Les paramètres UI demandent d’abord à `/v1/models`, puis valident le modèle sélectionné via une requête de chat légère. Si la découverte échoue mais que le chat fonctionne toujours, vérifiez si votre clé dispose des permissions model-list ou si un pare-feu bloque l’affichage de la liste des modèles.

Les modèles de raisonnement sont considérés comme un cas limite de protocole : Notemd fusionne le contenu des prompts système et utilisateur pour générer des identifiants de raisonnement OpenAI qui rejettent un rôle `system` distinct, et il expose `reasoningEffort` uniquement lorsque la définition du fournisseur le permet.

## Résolution de problèmes

- `401` ou `403` : vérifier la clé, l’accès au projet, l’état de facturation, et déterminer si le modèle sélectionné est activé pour cette clé.
- `404` sur le chat : la Base URL est généralement incorrecte ou inclut déjà `/chat/completions`.
- Le modèle apparaît dans la documentation mais pas dans le sélecteur : l’accès à model-list et l’accès au chat constituent des points de défaillance distincts ; vérifiez manuellement l’ID exact du modèle dans le profil du fournisseur.
- Le modèle de raisonnement rejette les messages : utilisez un ID de modèle couvert par le chemin de traitement des raisonnements OpenAI ou passez à un modèle de chat sans fonction de raisonnement.

## Quand l'utiliser

Utilisez OpenAI lorsque vous souhaitez le chemin hébergé par défaut le plus prévisible, lorsque la découverte via `/v1/models` est importante, ou lorsque un flux de travail dépend des sémantiques du modèle de raisonnement OpenAI. Préférez un gateway ou un fournisseur local lorsque le routage, la confidentialité ou les contrôles de coûts sont plus importants que le comportement direct de OpenAI API.
