# OpenAI 提供服務商

<TLDR>
**OpenAI 使用與 OpenAI 相容的共用傳輸方式。** 目前的預設值為 `gpt-4o`、`https://api.openai.com/v1`、`temperature: 0.5`，以及 `models-then-chat` 連線測試。當您需要精確的欄位語意時，請使用此頁面；若要進行類別層級的選擇，請參考 [LLM 提供商](/docs/providers/overview) 的概覽。

這是[Obsidian AI知識管理指南](/docs/pillar-ai-knowledge)的一部分。
</TLDR>

## 安裝

建立一個 OpenAI API 金鑰，在 Notemd 設定中加入 `OpenAI` 提供者設定檔，並保持預設的 Base URL，除非您刻意要連接到支援 OpenAI 的網關。模型欄位在一般用途時可保留為 `gpt-4o`，或改用您的帳戶能呼叫的其他 OpenAI 模型 ID。

運用專案特定的模型來控制成本：將更強大的模型用於研究、概念抽取及長篇內容的重構；則使用較便宜或速度較快的模型來處理翻譯、連結建議及簡短摘要。

## 端點與認證

| 欄位 | 目前的預設值 |
|-------|----------------|
| 運輸 | `openai-compatible` |
| API 金鑰 | 必填項目 |
| 基底 URL | `https://api.openai.com/v1` |
| 預設模型 | `gpt-4o` |
| 溫度 | `0.5` |
| 連線測試 | `/models`，接著是 `/chat/completions` |

Notemd會在附加 `chat/completions` 或 `models` 之前，將與 OpenAI 相容的基礎 URLs 進行標準化處理。在 Base URL 中不要包含尾隨的 `/chat/completions`，否則會造成路徑重複。

## 模型探索

OpenAI 使用通用的 OpenAI 相容型模型探索路徑。設定 UI 會先詢問 `/v1/models`，再透過簡易的聊天請求來驗證所選模型。如果模型探索失敗但聊天功能仍正常，請檢查您的金鑰是否具有模型列表的權限，或是是否有閘道阻擋了模型列表的顯示。

推理模型被視為協定中的邊緣情況： Notemd 會將系統與使用者的提示內容合併，以產生用於 OpenAI 推理的 ID，這些 ID 會拒絕單獨的 `system` 角色，且僅在供應商定義支援時才會暴露 `reasoningEffort`。

## 故障排除

- `401` 或 `403`：驗證金鑰、專案存取權、帳單狀態，以及所選模型是否已為該金鑰啟用。
- 聊天中的 `404`：Base URL 通常都是錯誤的，或者已經包含了 `/chat/completions`。
- 模型雖出現在文件中，但並不在選擇器中：model-list 存取與 chat 存取是兩個不同的故障點；請在供應商設定檔中手動測試正確的模型 ID。
- 推理模型拒絕處理訊息：請使用屬於 OpenAI 推理處理路徑的模型 ID，或改用非推理型聊天模型。

## 何時使用

當您需要最可預測的預設託管路徑、 `/v1/models` 的發現功能很重要，或工作流程取決於 OpenAI 推理模型語意時，請使用 OpenAI。若在路由、隱私或成本控制方面比直接的 OpenAI API 行為更為重要，則建議選用閘道或本地供應商。
