## Швидкий старт

1. **Встановіть і увімкніть**: отримайте плагін з Obsidian Marketplace.
2. **Налаштуйте LLM**: перейдіть до `Settings -> Notemd`, виберіть вашого провайдера LLM, наприклад OpenAI або локального провайдера на кшталт Ollama, і введіть API-ключ або URL.
3. **Відкрийте бічну панель**: натисніть на значок чарівної палички Notemd у лівій стрічці, щоб відкрити бічну панель.
4. **Обробіть нотатку**: відкрийте будь-яку нотатку й натисніть **"Process File (Add Links)"** на бічній панелі, щоб автоматично додати `[[wiki-links]]` до ключових понять.
5. **Запустіть швидкий робочий процес**: використайте типову кнопку **"One-Click Extract"**, щоб з одного місця послідовно запустити обробку, пакетну генерацію та очищення Mermaid.

Це все. Дослідіть налаштування, щоб увімкнути додаткові можливості, такі як веб-дослідження, переклад і генерація контенту.

## Підтримка мов

### Контракт мовної поведінки

| Аспект | Область | Типово | Примітки |
|---|---|---|---|
| `Мова інтерфейсу` | Лише текст інтерфейсу плагіна, наприклад налаштування, бічна панель, сповіщення та діалоги | `auto` | Наслідує мову Obsidian; поточні каталоги UI: `en`, `ar`, `de`, `es`, `fa`, `fr`, `id`, `it`, `ja`, `ko`, `nl`, `pl`, `pt`, `pt-BR`, `ru`, `th`, `tr`, `uk`, `vi`, `zh-CN`, `zh-TW`. |
| `Мова виводу задач` | Вивід завдань, згенерований LLM: посилання, підсумки, генерація, екстракція, ціль перекладу | `en` | Може бути глобальним або для кожного завдання окремо, якщо увімкнено `Використовувати різні мови для задач`. |
| `Вимкнути автопереклад` | Завдання, що не є Translate, зберігають контекст мови джерела | `false` | Явні завдання `Translate` усе одно примусово використовують налаштовану цільову мову. |
| Резервна локаль | Розв'язання відсутніх ключів UI | locale -> `en` | Підтримує стабільність UI, коли частина ключів ще не перекладена. |

- Підтримувані вихідні документи ведуться англійською та спрощеною китайською, а опубліковані переклади README наведені у шапці вище.
- Поточне покриття UI locale у застосунку точно відповідає явному каталогу в коді: `en`, `ar`, `de`, `es`, `fa`, `fr`, `id`, `it`, `ja`, `ko`, `nl`, `pl`, `pt`, `pt-BR`, `ru`, `th`, `tr`, `uk`, `vi`, `zh-CN`, `zh-TW`.
- Англійський fallback залишається страховкою реалізації, але підтримувані видимі поверхні покриті регресійними тестами й не повинні непомітно повертатися до англійської під час звичайного використання.
- Подальші деталі та рекомендації для внеску зібрані в [Мовному центрі](./docs/i18n/README.md).

## Можливості

### Обробка документів за допомогою ШІ
- **Підтримка Multi-LLM**: підключення до різних хмарних і локальних провайдерів LLM. Див. [Підтримувані провайдери LLM](#підтримувані-провайдери-llm).
- **Розумне chunking**: автоматично ділить великі документи на зручні частини за кількістю слів.
- **Збереження вмісту**: намагається зберегти оригінальне форматування, одночасно додаючи структуру й посилання.
- **Відстеження прогресу**: оновлення в реальному часі через бічну панель Notemd або модальне вікно прогресу.
- **Операції, які можна скасувати**: скасуйте будь-яке завдання обробки, одиночне чи пакетне, запущене з бічної панелі, окремою кнопкою скасування. Операції з палітри команд використовують модальне вікно, яке також можна скасувати.
- **Налаштування кількох моделей**: використовуйте різних провайдерів LLM і конкретні моделі для різних завдань, таких як Add Links, Research, Generate Title і Translate, або одного провайдера для всього.
- **Stable API Calls (Retry Logic)**: за потреби увімкніть автоматичні повторні спроби для невдалих викликів LLM API з налаштовуваними інтервалом і лімітом спроб.
- **Стійкіші тести з'єднання з провайдером**: якщо перша перевірка провайдера стикається з тимчасовим мережевим розривом, Notemd тепер переходить на стабільну послідовність повторів перед остаточною помилкою. Це охоплює шляхи OpenAI-compatible, Anthropic, Google, Azure OpenAI та Ollama.
- **Резервний транспорт залежно від середовища виконання**: якщо довгий запит до провайдера переривається всередині `requestUrl` через тимчасові мережеві помилки на кшталт `ERR_CONNECTION_CLOSED`, Notemd повторює ту саму спробу через резервний транспорт, залежний від середовища, ще до входу в налаштований цикл повторів. На desktop-збірках використовується Node `http/https`, а в не-desktop середовищах — браузерний `fetch`. Це зменшує кількість хибних збоїв на повільних gateway та reverse proxy.
- **Посилення ланцюжка стабільних довгих запитів для OpenAI-compatible**: у stable mode виклики OpenAI-compatible тепер використовують явний 3-етапний порядок для кожної спроби: primary direct streaming transport, потім direct non-stream transport, потім `requestUrl` fallback, який за потреби все ще може перейти на streamed parsing. Це зменшує кількість хибнонегативних ситуацій, коли провайдер завершує buffered response, але streaming pipe нестабільний.
- **Protocol-aware streaming fallback для всіх LLM API**: довгі fallback-спроби тепер переходять на streamed parsing з урахуванням протоколу в усіх вбудованих шляхах LLM, а не лише на OpenAI-compatible endpoint. Notemd тепер обробляє OpenAI/Azure-style SSE, Anthropic Messages streaming, відповіді Google Gemini SSE й Ollama NDJSON як на desktop `http/https`, так і на не-desktop `fetch`, а решта direct OpenAI-style provider entrypoint повторно використовують той самий спільний fallback-path.
- **Готові пресети для китайських провайдерів**: вбудовані пресети тепер охоплюють `Qwen`, `Qwen Code`, `Doubao`, `Moonshot`, `GLM`, `Z AI`, `MiniMax`, `Huawei Cloud MaaS`, `Baidu Qianfan` і `SiliconFlow` на додачу до наявних глобальних і локальних провайдерів.
- **Надійна пакетна обробка**: покращена логіка одночасної обробки завдяки **рознесеним у часі API-викликам**, щоб запобігати rate-limiting-помилкам і забезпечувати стабільну роботу під час великих пакетних задач. Нова реалізація гарантує, що завдання запускаються з різними інтервалами, а не всі одразу.
- **Точне звітування про прогрес**: виправлено помилку, через яку progress bar міг зависати, тому UI тепер завжди відображає реальний стан операції.
- **Надійна паралельна пакетна обробка**: усунуто проблему, через яку паралельні пакетні операції передчасно зупинялися, що забезпечує надійну й ефективну обробку всіх файлів.
- **Точність progress bar**: виправлено помилку, через яку progress bar для команди "Create Wiki-Link & Generate Note" зависав на 95%; тепер після завершення коректно відображається 100%.
- **Покращене API-debugging**: режим "API Error Debugging Mode" тепер захоплює повні response body від провайдерів LLM і пошукових сервісів, таких як Tavily і DuckDuckGo, а також записує transport timeline для кожної спроби з очищеними request URL, тривалістю, response headers, частковими response body, частково розібраним stream output і трасуванням стека для кращого усунення проблем на шляхах OpenAI-compatible, Anthropic, Google, Azure OpenAI та Ollama.
- **Панель режиму розробника**: налаштування тепер містять окрему діагностичну панель лише для розробників, яка прихована, доки не ввімкнено "Developer mode". Вона підтримує вибір діагностичного шляху виклику та повторні перевірки стабільності для вибраного режиму.
- **Оновлена бічна панель**: вбудовані дії згруповані в більш сфокусовані секції з яснішими мітками, живим статусом, прогресом із можливістю скасування і журналами, які можна копіювати, щоб зменшити захаращення бічної панелі. Нижній блок прогресу й журналів тепер залишається видимим навіть коли всі секції розгорнуті, а стан готовності використовує зрозумілішу доріжку очікування.
- **Поліпшення взаємодії та читабельності бічної панелі**: кнопки бічної панелі тепер мають чіткіший відгук на наведення, натискання та фокус, а кольорові CTA-кнопки, зокрема `One-Click Extract` і `Batch generate from titles`, мають кращий контраст тексту для читабельності в різних темах.
- **CTA лише для дій з одним файлом**: кольоровий стиль CTA тепер зарезервовано лише для дій над окремим файлом. Пакетні дії, дії над папками та змішані робочі процеси використовують не-CTA стиль, щоб зменшити помилки через неправильне сприйняття масштабу дії.
- **Користувацькі робочі процеси в один клік**: перетворюйте вбудовані інструменти бічної панелі на багаторазові користувацькі кнопки з власними назвами й ланцюжками дій. Типовий робочий процес `One-Click Extract` доступний одразу після встановлення.

### Розширення графа знань
- **Автоматичне wiki-linking**: визначає ключові поняття й додає `[[wiki-links]]` до оброблених нотаток на основі виводу LLM.
- **Створення concept note (необов'язково й налаштовувано)**: автоматично створює нові нотатки для виявлених понять у вказаній папці vault.
- **Налаштовувані вихідні шляхи**: окремо налаштуйте відносні шляхи всередині vault для збереження оброблених файлів і нових concept note.
- **Налаштовувані назви вихідних файлів для Add Links**: ви можете за бажанням **перезаписувати оригінальний файл** або використовувати власний suffix чи replacement string замість типового `_processed.md`, коли файли обробляються для додавання посилань.
- **Підтримка цілісності посилань**: базова підтримка оновлення посилань, коли нотатки змінюються або видаляються з vault.
- **Чисте виділення понять**: виділяйте поняття й створюйте відповідні concept note без зміни документа-джерела. Це зручно для наповнення бази знань із наявних документів без їх модифікації. Функція має налаштовувані опції мінімальних concept note та backlink.

### Переклад

- **Переклад за допомогою ШІ**:
  - Перекладайте вміст нотаток за допомогою налаштованого LLM.
  - **Підтримка великих файлів**: великі файли автоматично діляться на менші частини на основі параметра `Chunk word count` перед надсиланням до LLM. Потім перекладені фрагменти акуратно об'єднуються назад в один документ.
  - Підтримує переклад між багатьма мовами.
  - Налаштовувана цільова мова через settings або UI.
  - Автоматично відкриває перекладений текст праворуч від оригіналу для зручнішого читання.
- **Пакетний переклад**:
  - Перекладайте всі файли у вибраній папці.
  - Підтримує паралельну обробку, якщо ввімкнено "Enable Batch Parallelism".
  - Використовує користувацькі prompt для перекладу, якщо вони налаштовані.
  - Додає пункт "Batch translate this folder" у контекстне меню file explorer.
- **Вимкнути автоматичний переклад**: коли цю опцію ввімкнено, завдання, відмінні від Translate, більше не змушують результат бути певною мовою, зберігаючи контекст мови джерела. Явне завдання "Translate" все одно виконує переклад згідно з налаштуванням.

### Веб-дослідження та генерація контенту
- **Веб-дослідження та підсумовування**:
  - Виконуйте веб-пошук через Tavily, що потребує API key, або через DuckDuckGo, який лишається експериментальним.
  - **Покращена стійкість пошуку**: пошук DuckDuckGo тепер має покращену логіку парсингу, `DOMParser` з Regex fallback, щоб витримувати зміни верстки й повертати надійні результати.
  - Підсумовуйте результати пошуку за допомогою налаштованого LLM.
  - Мову підсумку можна налаштувати в settings.
  - Додавайте підсумки до поточної нотатки.
  - Налаштовуваний ліміт token для дослідницького контенту, що надсилається до LLM.
- **Генерація контенту з заголовка**:
  - Використовуйте заголовок нотатки, щоб згенерувати початковий контент через LLM, замінюючи наявний вміст.
  - **Необов'язкове дослідження**: налаштуйте, чи виконувати веб-дослідження через вибраного провайдера, щоб додати контекст для генерації.
- **Пакетна генерація контенту із заголовків**: генеруйте контент для всіх нотаток у вибраній папці на основі їхніх заголовків, з урахуванням необов'язкового параметра дослідження. Успішно оброблені файли переміщуються до **налаштовуваної підпапки "complete"**, наприклад `[foldername]_complete` або папки з власною назвою, щоб уникнути повторної обробки.
- **Зв'язка з Mermaid auto-fix**: коли Mermaid auto-fix увімкнено, Mermaid-related workflow тепер автоматично виправляють згенеровані файли або вихідні папки після обробки. Це охоплює Process, Generate from Title, Batch Generate from Titles, Research & Summarize, Summarise as Mermaid і Translate.

### Допоміжні можливості
- **Підсумувати як діаграму Mermaid**:
  - Ця функція дозволяє стисло подати вміст нотатки у вигляді Mermaid diagram.
  - Мову виходу Mermaid diagram можна налаштувати в settings.
  - **Mermaid Output Folder**: налаштуйте папку, куди зберігатимуться згенеровані Mermaid diagram files.
  - **Translate Summarize to Mermaid Output**: за бажанням перекладайте згенерований Mermaid-контент на налаштовану цільову мову.

  - **Deep Debug Mode**: якщо після первинного виправлення помилки лишаються, запускається просунутий режим deep debug. Він обробляє складні edge case, зокрема:
    - **Comment Integration**: автоматично зливає кінцеві коментарі, що починаються з `%`, в edge label, наприклад `A -- Label --> B; % Comment` перетворюється на `A -- "Label(Comment)" --> B;`.
    - **Malformed Arrows**: виправляє стрілки, поглинуті лапками, наприклад `A -- "Label -->" B` -> `A -- "Label" --> B`.
    - **Inline Subgraphs**: перетворює inline subgraph labels на edge labels.
    - **Reverse Arrow Fix**: виправляє нестандартні стрілки `X <-- Y` на `Y --> X`.
    - **Direction Keyword Fix**: забезпечує написання ключового слова `direction` малими літерами всередині subgraph, наприклад `Direction TB` -> `direction TB`.
    - **Comment Conversion**: перетворює коментарі `//` на edge labels, наприклад `A --> B; // Comment` -> `A -- "Comment" --> B;`.
    - **Duplicate Label Fix**: спрощує повторювані bracketed label, наприклад `Node["Label"]["Label"]` -> `Node["Label"]`.
    - **Invalid Arrow Fix**: перетворює невалідний синтаксис стрілки `--|>` на стандартний `-->`.
    - **Надійна обробка label і note**: покращена обробка міток зі спеціальними символами, як `/`, і краща підтримка користувацького синтаксису нотаток `note for ...`, що дозволяє чисто прибирати артефакти на кшталт зайвих дужок наприкінці.
    - **Advanced Fix Mode**: містить надійні виправлення для нецитованих node label, які містять пробіли, спеціальні символи або вкладені дужки, наприклад `Node[Label [Text]]` -> `Node["Label [Text]"]`, що забезпечує сумісність зі складними diagram на кшталт paths Stellar Evolution. Також виправляє пошкоджені edge labels, наприклад `--["Label["-->` у `-- "Label" -->`. Додатково перетворює inline comments, наприклад `Consensus --> Adaptive; # Some advanced consensus`, на `Consensus -- "Some advanced consensus" --> Adaptive`, а також виправляє незакриті лапки наприкінці рядка, замінюючи фінальне `;"` на `"]`.
    - **Note Conversion**: автоматично перетворює `note right/left of` і standalone comments `note :` на стандартні Mermaid node definition та connections, наприклад `note right of A: text` стає `NoteA["Note: text"]`, з'єднаним з `A`, що запобігає syntax error і покращує layout. Тепер підтримуються і arrow links (`-->`), і solid links (`---`).
    - **Extended Note Support**: автоматично перетворює `note for Node "Content"` і `note of Node "Content"` на стандартні зв'язані note node, наприклад `NoteNode[" Content"]`, з'єднану з `Node`, забезпечуючи сумісність із розширеним користувацьким синтаксисом.
    - **Enhanced Note Correction**: автоматично перейменовує нотатки з послідовною нумерацією, наприклад `Note1`, `Note2`, щоб уникнути aliasing-помилок, коли присутні кілька нотаток.
    - **Parallelogram/Shape Fix**: виправляє пошкоджені форми вузлів, такі як `[/["Label["/]`, до стандартного `["Label"]`, забезпечуючи сумісність зі згенерованим контентом.
    - **Standardize Pipe Labels**: автоматично виправляє й стандартизує edge label, що містять вертикальні риски, щоб вони були коректно взяті в лапки, наприклад `-->|Text|` стає `-->|"Text"|`, а `-->|Math|^2|` стає `-->|"Math|^2"|`.
    - **Misplaced Pipe Fix**: виправляє edge label, які помилково стоять перед стрілкою, наприклад `>|"Label"| A --> B` стає `A -->|"Label"| B`.
    - **Merge Double Labels**: виявляє та об'єднує складні подвійні label на одному edge, наприклад `A -- Label1 -- Label2 --> B` або `A -- Label1 -- Label2 --- B`, в одну чисту label із переносами рядка: `A -- "Label1<br>Label2" --> B`.
    - **Unquoted Label Fix**: автоматично бере в лапки node label, які містять потенційно проблемні символи, як лапки, знаки дорівнює або математичні оператори, але не мають зовнішніх лапок, наприклад `Plot[Plot "A"]` -> `Plot["Plot "A""]`, щоб запобігти render error.
    - **Intermediate Node Fix**: розділяє edge, які містять визначення проміжного node, на два окремі edge, наприклад `A -- B[...] --> C` стає `A --> B[...]` і `B[...] --> C`, забезпечуючи валідний Mermaid-синтаксис.
    - **Concatenated Label Fix**: надійно виправляє визначення node, де ID злипся з label, наприклад `SubdivideSubdivide...` стає `Subdivide["Subdivide..."]`, навіть якщо перед цим стоять pipe labels або повторення не є точним, шляхом валідації проти відомих node ID.

### Із Obsidian Marketplace (рекомендовано)
1. Відкрийте в Obsidian **Settings** -> **Community plugins**.
2. Переконайтеся, що "Restricted mode" **вимкнено**.
3. Натисніть **Browse** community plugins і знайдіть "Notemd".
4. Натисніть **Install**.
5. Після встановлення натисніть **Enable**.

### Ручне встановлення
1. Завантажте assets останнього release зі [сторінки GitHub Releases](https://github.com/Jacobinwwey/obsidian-NotEMD/releases). Кожен release також містить `README.md` для довідки в пакеті, але для ручного встановлення потрібні лише `main.js`, `styles.css` і `manifest.json`.
2. Перейдіть до папки конфігурації вашого Obsidian vault: `<YourVault>/.obsidian/plugins/`.
3. Створіть нову папку з назвою `notemd`.
4. Скопіюйте `main.js`, `styles.css` і `manifest.json` до папки `notemd`.
5. Перезапустіть Obsidian.
6. Перейдіть до **Settings** -> **Community plugins** і увімкніть "Notemd".

### Налаштування стабільних API-викликів
- **Увімкнути стабільні API-виклики (логіка повторів)**:
  - **Вимкнено (типово)**: один збій API-виклику зупинить поточне завдання.
  - **Увімкнено**: автоматично повторює невдалі виклики LLM API, що корисно при нестабільній мережі або обмеженнях за частотою.
  - **Резервний шлях для перевірки підключення**: навіть коли звичайні виклики ще не працюють у стабільному режимі, перевірки з'єднання з провайдером тепер переходять на ту саму послідовність повторів після першого тимчасового мережевого збою.
  - **Резервний транспорт під час виконання (з урахуванням середовища)**: довгі запити задач, тимчасово обірвані `requestUrl`, тепер повторюють ту саму спробу спочатку через резервний транспорт з урахуванням середовища. Desktop builds використовують Node `http/https`, а non-desktop environments — browser `fetch`. Ці резервні спроби тепер використовують потоковий розбір з урахуванням протоколу у вбудованих шляхах LLM, охоплюючи OpenAI-compatible SSE, Azure OpenAI SSE, Anthropic Messages SSE, Google Gemini SSE та Ollama NDJSON, завдяки чому повільні gateway можуть повертати body chunks раніше. Решта direct OpenAI-style provider entrypoint використовують той самий shared fallback path.
  - **Порядок стабільного режиму для OpenAI-compatible**: у stable mode кожна OpenAI-compatible attempt тепер проходить через `direct streaming -> direct non-stream -> requestUrl (with streamed fallback when needed)` перед тим, як вважатися невдалою. Це запобігає надто агресивним збоям, коли нестабільний лише один transport mode.
- **Інтервал повторів (секунди)**: видно лише коли функцію ввімкнено. Час очікування між повторними спробами, від 1 до 300 секунд. Типово: 5.
- **Максимальна кількість повторів**: видно лише коли функцію ввімкнено. Максимальна кількість повторних спроб, від 0 до 10. Типово: 3.
- **Режим налагодження помилок API**:
  - **Вимкнено (типово)**: використовує стандартне й лаконічне звітування про помилки.
  - **Увімкнено**: вмикає детальне логування помилок, подібне до verbose output у DeepSeek, для всіх провайдерів і завдань, включно з Translate, Search і Connection Tests. Це включає HTTP status codes, raw response text, request transport timelines, sanitized request URLs і headers, тривалість кожної attempt, response headers, partial response body, parsed partial stream output і stack traces, що критично для усунення проблем із підключенням до API та reset з боку upstream gateway.
- **Режим розробника**:
  - **Вимкнено (типово)**: ховає всі діагностичні елементи керування, призначені лише для розробників, від звичайних користувачів.
  - **Увімкнено**: показує окрему панель діагностики розробника в налаштуваннях.
- **Діагностика провайдера для розробника (довгий запит)**:

#### Параметри обробки
- **Enable Batch Parallelism**:
  - **Вимкнено (типово)**: пакетні завдання, такі як "Process Folder" або "Batch Generate from Titles", обробляють файли один за одним, послідовно.
  - **Увімкнено**: дозволяє плагіну обробляти кілька файлів одночасно, що може суттєво пришвидшити великі пакетні job.
- **Batch Concurrency**: видно лише коли ввімкнено parallelism. Визначає максимальну кількість файлів, які обробляються паралельно. Більше значення може працювати швидше, але споживає більше ресурсів і може впертися в API rate limits. Типово: 1, діапазон: 1-20.
- **Batch Size**: видно лише коли ввімкнено parallelism. Кількість файлів, що об'єднуються в один batch. Типово: 50, діапазон: 10-200.
- **Delay Between Batches (ms)**: видно лише коли ввімкнено parallelism. Необов'язкова затримка в мілісекундах між обробкою batch, що допомагає контролювати API rate limits. Типово: 1000 ms.
- **API Call Interval (ms)**: мінімальна затримка в мілісекундах *до і після* кожного окремого виклику LLM API. Критично для низьколімітних API або щоб запобігати 429-помилкам. Встановіть 0, якщо не хочете штучних затримок. Типово: 500 ms.
- **Chunk Word Count**: максимальна кількість слів у chunk, який надсилається до LLM. Впливає на кількість API calls для великих файлів. Типово: 3000.
- **Enable Duplicate Detection**: перемикає базову перевірку дублікатів слів у обробленому вмісті, результати виводяться в console. Типово: увімкнено.
- **Max Tokens**: максимальна кількість token, які LLM має згенерувати в одному response chunk. Впливає на вартість і деталізацію. Типово: 4096.

- **Tavily Max Results**: видно лише якщо вибрано Tavily. Максимальна кількість search results, які має повернути Tavily, від 1 до 20. Типово: 5.
- **Tavily Search Depth**: видно лише якщо вибрано Tavily. Виберіть `basic`, типово, або `advanced`. Примітка: `advanced` дає кращі результати, але коштує 2 API credits за пошук замість 1.
- **DuckDuckGo Max Results**: видно лише якщо вибрано DuckDuckGo. Максимальна кількість search results для парсингу, від 1 до 10. Типово: 5.
- **DuckDuckGo Content Fetch Timeout**: видно лише якщо вибрано DuckDuckGo. Максимальна кількість секунд очікування під час спроби отримати контент із кожного URL результату DuckDuckGo. Типово: 15.
- **Max Research Content Tokens**: приблизна максимальна кількість token з об'єднаних web research results, snippets або fetched content, яка включатиметься до prompt підсумовування. Допомагає контролювати розмір context window і вартість. Типово: 3000.

- **Enable Focused Learning Domain**:
  - **Вимкнено (типово)**: prompt, що надсилаються до LLM, використовують стандартні інструкції загального призначення.
  - **Увімкнено**: дозволяє вказати одну чи кілька галузей знань, щоб покращити контекстуальне розуміння LLM.
- **Learning Domain**: видно лише коли попередню опцію ввімкнено. Введіть свою конкретну галузь або галузі, наприклад 'Materials Science', 'Polymer Physics', 'Machine Learning'. Це додасть рядок "Relevant Fields: [...]" на початок prompt, допомагаючи LLM генерувати точніші й релевантніші links і контент для вашої предметної області.

## Посібник з використання

### Швидкі робочі процеси й бічна панель

- Відкрийте бічну панель Notemd, щоб отримати доступ до згрупованих секцій дій для основної обробки, генерації, перекладу, знань і утиліт.
- Використовуйте область **Швидкі робочі процеси** у верхній частині бічної панелі, щоб запускати власні багатокрокові кнопки.
- Типовий робочий процес **One-Click Extract** запускає `Process File (Add Links)` -> `Batch Generate from Titles` -> `Batch Mermaid Fix`.
- Прогрес робочого процесу, логи кожного кроку та помилки відображаються в бічній панелі, а закріплений нижній блок не дає progress bar і log area бути витісненими розгорнутими секціями.
- Картка прогресу залишає текст стану, окрему плашку з відсотком і час, що залишився, читабельними з першого погляду, а ті самі власні робочі процеси можна переналаштувати з налаштувань.

   - Плагін прочитає файли, визначить поняття і створить для них нові нотатки у вашій **Concept Note Folder**, не змінюючи оригінальні файли.

8. **Create Wiki-Link & Generate Note from Selection**:
   - Ця потужна команда спрощує процес створення й наповнення нових concept note.
   - Виділіть слово або фразу в editor.
   - Запустіть команду `Notemd: Create Wiki-Link & Generate Note from Selection`; рекомендовано призначити їй hotkey, наприклад `Cmd+Shift+W`.
   - Плагін виконає:
     1. Замінить виділений текст на `[[wiki-link]]`.
     2. Перевірить, чи існує нотатка з таким заголовком у **Concept Note Folder**.
     3. Якщо існує, додасть backlink до поточної нотатки.
     4. Якщо не існує, створить нову порожню нотатку.
     5. Потім автоматично запустить команду **"Generate Content from Title"** для нової або наявної нотатки й наповнить її AI-згенерованим контентом.

9. **Extract Concepts and Generate Titles**:
   - Ця команда об'єднує дві потужні функції в один зручний робочий процес.
   - Запустіть команду `Notemd: Extract Concepts and Generate Titles` з палітри команд; рекомендовано призначити їй hotkey.
   - Плагін виконає:
     1. Спочатку завдання **"Extract concepts (current file)"** для поточного активного файлу.
     2. Потім автоматично запустить завдання **"Batch generate from titles"** для папки, яку ви вказали в settings як **Concept note folder path**.
   - Це дозволяє спочатку поповнити базу знань новими поняттями з вихідного документа, а потім одразу розгорнути ці нові concept note AI-згенерованим контентом в одному кроці.

10. **Extract Specific Original Text**:
    - Налаштуйте свої питання в settings у секції "Extract Specific Original Text".
    - Використовуйте кнопку "Extract Specific Original Text" у sidebar, щоб обробити активний файл.
    - **Merged Mode**: дозволяє швидшу обробку, надсилаючи всі питання в одному prompt.
    - **Translation**: за бажанням перекладає витягнутий текст обраною мовою.
    - **Custom Output**: налаштовуйте, де і як зберігається файл із витягнутим текстом.

11. **Batch Mermaid Fix**:
    - Використовуйте кнопку "Batch Mermaid Fix" у sidebar, щоб просканувати папку й виправити типові Mermaid syntax errors.
    - Плагін повідомить про файли, у яких після цього все ще лишилися помилки, у файлі `mermaid_error_{foldername}.md`.
    - За бажанням налаштуйте плагін так, щоб переміщувати ці проблемні файли до окремої папки для review.

## Використання мережі та обробка даних

Notemd працює локально всередині Obsidian, але деякі функції надсилають зовнішні запити.

### Виклики до провайдерів LLM (налаштовувані)

- Тригер: обробка файлів, генерація, переклад, підсумування досліджень, Mermaid-підсумування та actions з перевірки підключення або діагностики.
- Endpoint: ваші налаштовані provider base URL у settings Notemd.
- Дані, що надсилаються: текст prompt і task content, потрібний для обробки.
- Примітка щодо обробки даних: API keys налаштовуються локально в settings плагіна і використовуються для підпису запитів з вашого пристрою.

### Виклики веб-дослідження (необов'язково)

- Тригер: коли веб-дослідження ввімкнено і вибрано search provider.
- Endpoint: Tavily API або DuckDuckGo endpoint.
- Дані, що надсилаються: ваш пошуковий запит і необхідні request metadata.

### Діагностика для розробників і журнали налагодження (необов'язково)

- Тригер: режим API debug і дії developer diagnostic.
- Зберігання: diagnostic і error log записуються в корінь вашого vault, наприклад `Notemd_Provider_Diagnostic_*.txt` і `Notemd_Error_Log_*.txt`.
- Примітка про ризики: logs можуть містити уривки request і response. Переглядайте їх перед публічним поширенням.

### Локальне зберігання

- Конфігурація плагіна зберігається в `.obsidian/plugins/notemd/data.json`.
- Згенеровані файли, звіти й необов'язкові log зберігаються у вашому vault згідно з settings.

## Усунення неполадок

### Поширені проблеми
- **Плагін не завантажується**: переконайтеся, що `manifest.json`, `main.js` і `styles.css` лежать у правильній папці, `<Vault>/.obsidian/plugins/notemd/`, і перезапустіть Obsidian. Перевірте Developer Console через `Ctrl+Shift+I` або `Cmd+Option+I`, чи немає помилок під час startup.
- **Збої обробки / API errors**:
  1. **Перевірте формат файлу**: переконайтеся, що файл, який ви намагаєтеся обробити або перевірити, має розширення `.md` або `.txt`. Наразі Notemd підтримує лише ці текстові формати.
  2. Використайте команду або кнопку "Test LLM Connection", щоб перевірити settings для активного провайдера.
  3. Ще раз перевірте API Key, Base URL, Model Name і API Version, для Azure. Переконайтеся, що API key правильний і має достатньо credits або permissions.
  4. Переконайтеся, що ваш локальний LLM server, LMStudio або Ollama, запущений, а Base URL правильний, наприклад `http://localhost:1234/v1` для LMStudio.
  5. Перевірте інтернет-з'єднання для хмарних провайдерів.
  6. **Для помилок обробки одного файлу:** перегляньте Developer Console для детальних error message. За потреби скопіюйте їх кнопкою в error modal.
  7. **Для пакетних помилок:** перевірте файл `error_processing_filename.log` у корені vault, щоб побачити детальні error message для кожного файлу, що завершився невдало. Developer Console або error modal можуть показувати лише summary або загальну batch error.
  8. **Автоматичні журнали помилок:** якщо процес завершується невдачею, плагін автоматично зберігає детальний log-файл із назвою `Notemd_Error_Log_[Timestamp].txt` у кореневому каталозі vault. Цей файл містить error message, трасування стека і session logs. Якщо проблеми повторюються, перевірте цей файл. Увімкнення "API Error Debugging Mode" у settings наповнить log ще детальнішими API response data.
  9. **Діагностика довгих запитів до реального endpoint, для розробників**:
     - Шлях усередині плагіна, рекомендовано спочатку: використайте **Settings -> Notemd -> Developer provider diagnostic (long request)**, щоб запустити runtime probe на активному провайдері й згенерувати `Notemd_Provider_Diagnostic_*.txt` у корені vault.
     - CLI-шлях, поза Obsidian runtime: для відтворюваного порівняння buffered і streaming behavior на рівні endpoint використайте:
       ```bash
       npm run diagnose:llm -- \
         --transport openai-compatible \
         --provider-name OpenRouter \
         --base-url https://openrouter.ai/api/v1 \
         --api-key "$OPENROUTER_API_KEY" \
         --model anthropic/claude-3.7-sonnet \
         --prompt-file ./tmp/prompt.txt \
         --content-file ./tmp/content.txt \
         --mode compare \
         --timeout-ms 360000 \
         --output ./tmp/openrouter-diagnostic.txt
       ```
       Згенерований звіт містить timing для кожної attempt, тобто `First Byte` і `Duration`, очищені request metadata, response headers, raw або partial body fragments, parsed stream fragments і точки збоїв transport layer.
- **Проблеми з підключенням LM Studio/Ollama**:
  - **Перевірка з'єднання не проходить**: переконайтеся, що локальний сервер, LM Studio або Ollama, запущений і правильну model завантажено або вона доступна.
  - **CORS Errors, Ollama on Windows**: якщо ви стикаєтеся з CORS, тобто Cross-Origin Resource Sharing errors, під час використання Ollama у Windows, вам може знадобитися задати environment variable `OLLAMA_ORIGINS`. Це можна зробити, виконавши `set OLLAMA_ORIGINS=*` у command prompt перед запуском Ollama. Це дозволяє запити з будь-якого origin.
  - **Enable CORS in LM Studio**: для LM Studio ви можете увімкнути CORS безпосередньо в server settings; це може бути необхідно, якщо Obsidian працює в browser або має суворі origin policies.
- **Помилки створення папок ("File name cannot contain...")**:
  - Зазвичай це означає, що шлях, указаний у settings, тобто **Processed File Folder Path** або **Concept Note Folder Path**, є недійсним *для Obsidian*.
  - **Переконайтеся, що ви використовуєте відносні шляхи**, наприклад `Processed` або `Notes/Concepts`, а **не абсолютні шляхи**, як `C:\Users\...` або `/Users/...`.
  - Перевірте недійсні символи: `* " \ / < > : | ? # ^ [ ]`. Зверніть увагу, що `\` є недійсним навіть у Windows для шляхів Obsidian. Використовуйте `/` як path separator.
- **Проблеми з продуктивністю**: обробка великих файлів або багатьох файлів може займати час. Зменште параметр "Chunk Word Count", якщо хочете потенційно швидші, але численніші API calls. Спробуйте іншого провайдера LLM або іншу модель.
- **Неочікуване linking**: якість linking сильно залежить від LLM і prompt. Експериментуйте з різними моделями або параметрами температури.
