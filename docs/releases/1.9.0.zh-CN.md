# Notemd v1.9.0

## 亮点

- **高级 batch 选择终于具备 run-safe 语义**：开发者开关控制的文件夹任务弹窗现已支持已保存筛选档案、单次文件夹/规则覆盖、preset chips，以及执行前的内联匹配文件预览。
- **放开输入文件限制现在是真正的能力契约，不再是假开关**：部分保留原文件的任务可读取更广泛的文本类文件，以及通过 Obsidian PDF 运行时提取文本的 PDF；而改写原文和逐字敏感流程仍继续保持严格限制。
- **`1.9.0` 发布真值已同步完成**：随包版本元数据、欢迎弹窗最近两次更新摘要、README 标记、chronicle 刷新与双语 release notes 已统一对齐到当前发布边界。

## 新功能

- 新增开发者开关 `放开输入文件限制`。
- 新增共享 supported-input reader 层，为 allowlist 任务提供更广泛文本类文件与 PDF 文本读取能力。
- 扩展高级 batch 文件夹弹窗，补齐：
  - 已保存文件筛选档案
  - 当前默认筛选的虚拟档案
  - 不修改已保存档案的单次文件夹覆盖
  - 单次规则覆盖与可点击 preset chips
  - 支持滚动检查的匹配文件预览面板

## 修复与鲁棒性

- 侧边栏动作、命令面板入口、batch host adapter 与按路径图形生成入口现在复用同一套任务级输入校验契约，不再各自漂移。
- 在 relaxed mode 下，批量翻译与批量提取概念的“无匹配文件”提示已改为 `受支持的输入文件` 语义，不再错误地写死成“没有 `.md` / `.txt` 文件”。
- PDF 输入现在通过 Obsidian PDF.js 运行时与二进制读取链路处理，避免只放开扩展名却不修改读取路径的伪支持。
- Developer-only 的 batch 高级复杂度仍然要求同时开启 `Developer mode` 与 `Advanced batch file selection`，普通用户默认继续走更简单的 legacy 选择流程。

## 对比验证摘要

- **改动前**：如果只放开扩展名，后续 reader 仍按纯文本假设运行，新的输入类型很容易在运行时失败。
- **改动后**：通过任务 allowlist 与共享 reader 层，放开输入类型的边界变成显式契约，而高风险任务继续锁定在稳定文件类别上。
- **改动前**：已保存 regex/glob 档案虽然能力增强，但运行前仍缺少足够的单次微调与可见性护栏。
- **改动后**：batch 执行前可复用已保存档案、临时覆盖文件夹或筛选规则、直接预览命中的文件列表，同时保持原保存档案不被污染。
- **验证**：发布前已在本机串行跑完 repo-saga 刷新，并完成 `npm run build`、`npm test -- --runInBand`、`npm run audit:i18n-ui`、`npm run audit:render-host`、`obsidian help`、`obsidian-cli help`、repo-local `obsidian build`、repo-local `obsidian test` 与 `git diff --check`。
