# Notemd v1.9.1

## 亮点

- **本次发布覆盖完整 `1.9.0 -> 1.9.1` 累积区间**：以下内容基于真实的 `git diff 1.9.0..HEAD` 生成，而不是只描述最后一个 patch 或最后一次 provider-settings 提交。
- **Provider 配置更简洁但没有损失能力**：默认只显示必要连接字段，同时保留并可继续使用已有高级覆盖项。
- **模型发现能力更广且边界更安全**：`获取模型列表` 现在支持更多 provider catalog 形态，但发现到的输出 token 上限只会写入当前 provider 的 override，不会改写全局运行时 token 预算。
- **token 上限 fallback 现在是 fail-closed**：当上游 catalog 无法提供可信的 max-output-tokens 时，Notemd 会优先保留已有有效 provider override，只在没有有效覆盖值时才走 fallback，并明确提示用户人工复核。

## 主要升级

- 扩展了有界 provider model discovery，覆盖更多 OpenAI-compatible endpoint family、托管网关变体、额外 response envelope、分页/continuation 形态，以及 provider-specific catalog 格式。
- 保持 provider settings 面板采用 metadata-driven 的核心/高级分层，同时继续保留已保存的高级 provider 状态，并让高级区块在多次打开设置时保持一致行为。
- 强化了 model-aware token guidance，让用户能明确区分：
  - provider 级输出 token override
  - 全局 `最大Token数`
  - 推荐的 `分块词数`
- 保留当前“手动编辑模型”仍可驱动全局自动管理 token 基线的行为，同时把“发现模型并应用”的能力明确限制在 provider override 通道。
- 扩展了 Stage-C 本地知识库工作流，使任务级知识库来源、inspect/evaluation 可见性，以及失败/部分可用场景下的行为更加明确。
- 加强了图形维护者验证链路的 semantic gate 覆盖，继续让 renderer-boundary 与 packaging-boundary 检查对齐当前单入口运行时契约。
- 继续保持 packaging 在潜在 render-host runtime lane 上 fail-closed，避免当前 `main` 再次漂移回“半交付的 detached runtime topology”。
- 保留了有界的 maintainer CLI/public-surface 路径，包括 repo-local `obsidian` / `obsidian-cli` helper 对齐，以及检入式 release-helper 契约覆盖。
- 重新同步了面向发布的真值面：版本元数据、欢迎弹窗最近两次更新摘要、双语 release notes、changelog 与 README 家族标记现在都已对齐到 `1.9.1`。

## 修复与鲁棒性

- 修复了 discovered-model apply 会把模型输出上限误当作全局 `最大Token数` 同步的问题。
- 修复了上游无法解析 discovered-model ceiling 时会清空或降级已有 provider override 的问题。
- 修复了欢迎弹窗近期更新摘要的版本新鲜度，现在会按本地语言显示 `1.9.1` 与 `1.9.0` 的正确摘要。
- 修复了 provider 高级区块的持久化行为，避免已有非默认高级设置在重新打开设置后被错误折叠而难以发现。
- 修复了欢迎弹窗 release notes 的本地化选择逻辑，避免继续回退到过时的英文摘要。
- 将进度文档与方案文档重新对齐到当前已交付的 provider-settings 架构，特别是 provider override、有界模型发现与 token fallback 语义。

## 对比摘要

- **改动前**：模型发现与 token guidance 之间仍有一个模糊边界，用户容易把 discovered-model apply 理解为“同步全局最大Token数”的路径。
- **改动后**：已交付行为现在是明确且有界的。模型发现只会把 ceiling 应用到当前 provider override；全局运行时预算仍由手动模型感知跟踪控制，除非用户主动偏离。
- **改动前**：上游 token ceiling 缺失时，已有有效 provider-specific override 可能被更弱的 fallback 覆盖。
- **改动后**：fallback 仅作为最后手段存在。如果已有有效 provider override，则始终保留；否则才使用保守默认值，并明确要求用户人工检查。
