# Notemd v1.9.1

## Highlights

- **This release covers the full accumulated `1.9.0 -> 1.9.1` delta**: the notes below are derived from the actual `git diff 1.9.0..HEAD` range instead of only the last patch or the final provider-settings commit.
- **Provider setup is now simpler without losing power**: required connection fields stay visible by default, while existing advanced overrides remain preserved and discoverable.
- **Model discovery is broader and safer**: fetch-model-list now supports more provider catalog shapes, but discovered output-token ceilings stay scoped to the active provider override instead of mutating the global runtime token budget.
- **Token-cap fallback now fails closed**: when an upstream catalog cannot provide a trustworthy max-output-tokens value, Notemd preserves an existing valid provider override and only falls back with explicit manual-review guidance.

## Major Upgrades

- Expanded bounded provider model discovery across more OpenAI-compatible endpoint families, hosted gateway variants, additional response envelopes, continuation patterns, and provider-specific catalog formats.
- Kept the provider settings panel on a metadata-driven core/advanced split, while preserving saved advanced provider state and making the advanced section behavior consistent across settings reopen cycles.
- Strengthened model-aware token guidance so users can see the boundary between:
  - provider-scoped output-token override
  - global `Max tokens`
  - recommended `Chunk word count`
- Preserved the current manual-model-edit behavior that can still drive the auto-managed global token baseline, while isolating discovered-model apply to the provider override lane.
- Extended Stage-C local knowledge workflows so task-scoped knowledge-base sources and inspect/evaluation surfaces behave more explicitly under failure and partial-availability conditions.
- Tightened the diagram-maintainer verification lane with deeper semantic gate coverage, keeping renderer-boundary and packaging-boundary checks aligned with the current single-entry runtime contract.
- Kept packaging fail-closed around the latent render-host runtime lane so current `main` does not accidentally drift back into a half-shipped detached-runtime topology.
- Preserved the bounded maintainer CLI/public-surface path, including repo-local `obsidian`/`obsidian-cli` helper alignment and checked-in release-helper contract coverage.
- Re-synced release-facing truth surfaces: version metadata, welcome modal latest-two digest, bilingual release notes, changelog, and README family markers all now align to `1.9.1`.

## Fixes And Robustness

- Fixed discovered-model apply so the selected model's output ceiling no longer behaves like a silent rewrite of global `Max tokens`.
- Fixed unresolved discovered-model ceiling handling so existing valid provider overrides are preserved instead of being cleared or downgraded.
- Fixed release-digest freshness so the welcome modal now shows `1.9.1` plus `1.9.0` with locale-appropriate highlights.
- Fixed provider advanced-section persistence so saved non-default advanced settings do not disappear behind a collapsed panel after reopening settings.
- Fixed localized release-note selection so the shipped welcome digest stays aligned with the user's UI locale instead of defaulting to stale English copy.
- Realigned progress and planning docs with the current shipped provider-settings architecture, especially around provider overrides, bounded discovery, and token-cap fallback semantics.

## Comparative Summary

- **Before**: model discovery and token guidance still had one ambiguous edge where discovered-model apply could be read as a global max-token sync path.
- **After**: the shipped behavior is explicit and bounded. Discovery applies model ceilings to the active provider override, while the global runtime budget remains governed by manual model-aware tracking unless the user chooses to diverge.
- **Before**: unresolved upstream token ceilings risked flattening a valid provider-specific override into a weaker fallback path.
- **After**: fallback is now last-resort only. If a valid provider override already exists, it wins; otherwise Notemd applies a conservative default and tells the user to review it manually.
